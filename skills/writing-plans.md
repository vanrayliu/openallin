# Writing Plans — 计划编写

> 当你有多步任务的规格或需求时，在编写代码之前使用此技能。

## 核心原则

1. **原子性** — 每个任务应该足够小，可以在一次会话中完成
2. **独立性** — 任务之间应该尽可能独立
3. **可验证** — 每个任务完成后应该有明确的验证标准
4. **有序性** — 任务应该按依赖关系排序

## 核心设计：多轮迭代

```
┌─────────────────────────────────────────────────────────────┐
│  规划是多轮迭代过程：                                       │
│                                                             │
│  草案 → 用户反馈 → 修改 → 用户确认 → 定稿                   │
│    ↑                                      │                │
│    └──────────────────────────────────────┘                │
│                                                             │
│  每轮向用户展示草案，直到确认通过                            │
└─────────────────────────────────────────────────────────────┘
```

## 每轮工作流

### 轮次结构

```
第 N 轮
━━━━━━━━
1. 展示草案
2. 用户反馈
3. 修改草案
4. 用户确认（继续/修改/通过）
```

### 第 1 轮：理解需求
```
目标：理解要做什么

1. 收集信息
   - 变更目标是什么？
   - 影响范围有多大？
   - 有哪些约束条件？

2. 初步拆分
   - 识别主要模块
   - 识别关键任务
   - 识别明显依赖

3. 输出草案 v0.1
   - 任务列表初稿
   - 初步的波次划分
   - 待确认问题清单
```

### 第 2 轮：完善拆分
```
目标：根据用户反馈完善任务拆分

1. 处理用户反馈
   - 用户补充了哪些信息？
   - 用户修改了哪些内容？

2. 细化任务
   - 每个任务是否足够小？
   - 任务之间依赖是否正确？
   - 验证标准是否明确？

3. 输出草案 v0.2
   - 任务列表更新版
   - 更清晰的波次划分
```

### 第 3 轮：风险评估
```
目标：识别风险和依赖

1. 识别风险
   - 技术风险有哪些？
   - 需要人工确认的决策？
   - 回滚方案是什么？

2. 完善波次
   - Wave 1: 基础/独立任务
   - Wave 2: 依赖 Wave 1 的任务
   - Wave 3: 集成/测试任务

3. 输出草案 v0.3
   - 完整任务列表
   - 波次计划
   - 风险清单
```

### 第 4 轮：最终确认
```
目标：获得用户对计划的最终确认

1. 展示完整计划
   - 所有任务和依赖
   - 波次执行顺序
   - 风险和缓解方案

2. 用户逐项确认
   - 每个任务是否合理？
   - 验收标准是否正确？
   - 优先级是否正确？

3. 定稿
   - 输出最终计划
   - 保存为执行依据
```

## 轮次总结格式

每轮结束时输出：

```markdown
## 第 N 轮总结

### 📋 草案版本
- 版本: v0.N
- 任务数: N
- 波次数: N

### ✅ 已确认
- ...

### ❓ 待确认
- ...

### 📝 用户反馈
- ...

### 用户决策
[ ] 继续下一轮
[ ] 基本 OK，细节可以后续调整
[ ] 需要大幅修改
[ ] 计划通过，进入执行
```

## 状态保存

每轮结束后保存到 `workspace/PLAN.md`：

```markdown
# 执行计划

> 日期: YYYY-MM-DD
> 版本: v0.N
> 状态: 草稿/确认中/已定稿

## 概述

### 变更目标
- ...

### 影响范围
- ...

### 技术选型
- ...

## 任务列表

### Wave 1（可并行）
- [ ] Task 1: ...
- [ ] Task 2: ...

### Wave 2
- [ ] Task 3: ...

### Wave 3
- [ ] Task 4: ...

| ... | ... | ... |

### 第 1 轮：...

```

## 终止条件

满足以下任一条件，规划结束：

1. **用户确认** — 用户说"计划 OK，开始执行"
2. **最大轮次** — 完成 4 轮后强制定稿
3. **用户决策** — 用户跳过规划直接执行

## 任务拆分原则

### 好的任务
- 可以在 30 分钟内完成
- 有明确的输入和输出
- 可以独立验证
- 不依赖其他未完成的任务

### 坏的任务
- "实现整个系统"
- "优化性能"
- "修复所有 bug"
- 模糊不清的描述

## 输出物

1. **任务清单** — `workspace/PLAN.md`
2. **执行波次** — 任务分组和执行顺序
3. **风险清单** — 风险和缓解方案
4. **验收标准** — 每个任务的完成标准
