# Writing Plans — 计划编写

> 当你有多步任务的规格或需求时，在编写代码之前使用此技能。

## 核心原则

1. **原子性** — 每个任务应该足够小，可以在一次会话中完成
2. **独立性** — 任务之间应该尽可能独立
3. **可验证** — 每个任务完成后应该有明确的验证标准
4. **有序性** — 任务应该按依赖关系排序

## 计划结构

### 1. 概述
- 变更目标
- 影响范围
- 技术选型

### 2. 任务分解
```
## Task 1: <任务名>
- 描述: <做什么>
- 文件: <涉及哪些文件>
- 验证: <如何确认完成>
- 依赖: <前置任务>

## Task 2: ...
```

### 3. 执行顺序
- Wave 1: 独立任务（可并行）
- Wave 2: 依赖 Wave 1 的任务
- Wave 3: 集成任务

### 4. 风险点
- 可能的技术风险
- 需要人工确认的决策
- 回滚方案

## 任务拆分原则

### 好的任务
- 可以在 30 分钟内完成
- 有明确的输入和输出
- 可以独立验证
- 不依赖其他未完成的任务

### 坏的任务
- "实现整个系统"
- "优化性能"
- "修复所有 bug"
- 模糊不清的描述

## 输出物

- 任务清单（tasks.md）
- 执行波次计划
- 风险清单
