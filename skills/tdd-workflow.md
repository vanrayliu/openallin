# TDD Workflow — 测试驱动开发

> 在实现任何功能或修复任何 bug 之前使用此技能。

## 核心原则

1. **红** — 先写一个失败的测试
2. **绿** — 用最简单的代码让测试通过
3. **重构** — 优化代码，保持测试通过

## 工作流

### Step 1: 理解需求
- 明确要实现的行为
- 识别输入、输出、边界条件

### Step 2: 写失败的测试（Red）
```
1. 编写测试用例
2. 运行测试 → 应该失败
3. 确认失败原因正确
```

### Step 3: 实现最小代码（Green）
```
1. 编写刚好能让测试通过的代码
2. 运行测试 → 应该通过
3. 不要多做，只做让测试通过的事
```

### Step 4: 重构（Refactor）
```
1. 优化代码结构
2. 消除重复
3. 运行测试 → 必须全部通过
```

### Step 5: 重复
- 回到 Step 2，写下一个测试
- 逐步完善功能

## 测试优先级

1. **核心业务逻辑** — 必须测试
2. **边界条件** — 必须测试
3. **错误处理** — 必须测试
4. **UI 组件** — 按需测试
5. **纯工具函数** — 按需测试

## 注意事项

- 测试是文档，写可读的测试
- 一个测试只验证一个行为
- 测试名描述行为，不是实现
- 不要测试私有实现细节
