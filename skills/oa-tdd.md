# /oa-tdd — 测试驱动开发

当用户输入 `/oa-tdd <feature>` 时执行此技能。

## 核心原则

**测试驱动开发（Test-Driven Development）** 是一种先写测试再写代码的开发方法：

```
┌─────────────────────────────────────────────┐
│  Red → Green → Refactor → Repeat            │
│                                             │
│  1. Red:    先写一个失败的测试               │
│  2. Green:  写最少的代码让测试通过           │
│  3. Refactor: 重构代码，保持测试通过         │
│  4. Repeat: 重复以上步骤                     │
└─────────────────────────────────────────────┘
```

## 工作流程

### 第 1 步：理解需求

- 用户想要实现什么功能？
- 输入是什么？输出是什么？
- 边界条件是什么？

### 第 2 步：编写失败测试（Red）

```markdown
1. 创建测试文件（如果不存在）
2. 编写测试用例描述预期行为
3. 运行测试，确认测试失败
4. 输出：测试文件路径、失败原因
```

**测试命名规范：**
```
test_<功能>_<场景>.py          # Python
<功能>.test.<场景>.js          # JavaScript
describe('<功能>', () => {     # Jest
  it('should <场景>', () => {});
});
```

### 第 3 步：编写最小实现（Green）

```markdown
1. 只写让测试通过的最少代码
2. 不要提前优化
3. 不要添加额外功能
4. 运行测试，确认测试通过
```

### 第 4 步：重构（Refactor）

```markdown
1. 检查代码是否有重复
2. 检查代码是否清晰易懂
3. 检查命名是否准确
4. 重构后运行测试，确认仍然通过
```

### 第 5 步：重复

```markdown
1. 下一个测试用例是什么？
2. 重复 Red → Green → Refactor
3. 直到所有功能完成
```

## 输出格式

每轮 TDD 循环输出：

```markdown
## TDD 循环 #N

### 🔴 Red
- 测试文件：`<path>`
- 测试内容：`<description>`
- 运行结果：❌ 失败（预期）

### 🟢 Green
- 实现文件：`<path>`
- 实现代码：`<code snippet>`
- 运行结果：✅ 通过

### 🔵 Refactor
- 重构内容：`<changes>`
- 运行结果：✅ 通过

### 下一步
- 继续下一个测试用例？还是完成？
```

## 使用场景

- 实现新功能
- 修复 Bug（先写失败的测试重现 Bug）
- 重构代码（确保有测试覆盖）

## 注意事项

1. **测试先行** - 必须先写测试，再写实现
2. **最小实现** - 只写让测试通过的代码，不多写
3. **频繁运行** - 每次修改后都运行测试
4. **小步前进** - 每次只添加一个测试用例
5. **保持简单** - 不要过度设计

## 示例

```
用户: /oa-tdd 用户登录功能

AI: 
## 理解需求
- 输入：用户名、密码
- 输出：登录成功/失败
- 边界：用户名不存在、密码错误、空输入

## TDD 循环 #1

### 🔴 Red
- 测试文件：`tests/auth/login.test.js`
- 测试内容：正确用户名密码应该登录成功
- 运行结果：❌ 失败（login 函数不存在）

### 🟢 Green
- 实现文件：`src/auth/login.js`
- 实现代码：
  ```javascript
  function login(username, password) {
    return { success: true };
  }
  ```
- 运行结果：✅ 通过

### 下一步
- 继续添加测试用例：用户名不存在的情况？
```
