# Systematic Debugging — 系统化调试

> 遇到任何 bug、测试失败或意外行为时，在提出修复方案之前使用此技能。

## 核心原则

1. **不要猜测** — 先理解，再修复
2. **复现问题** — 不能复现的 bug 无法修复
3. **定位根因** — 修复症状 ≠ 修复问题
4. **验证修复** — 修复后必须验证

## 核心设计：多轮迭代

```
┌─────────────────────────────────────────────────────────────┐
│  调试是多轮迭代过程：                                       │
│                                                             │
│  假设 → 验证 → 定位 → 修复 → 测试                          │
│    ↑                                      │                │
│    └──────────────────────────────────────┘                │
│                                                             │
│  每轮结束后向用户确认，直到 bug 完全解决                     │
└─────────────────────────────────────────────────────────────┘
```

## 每轮工作流

### 轮次结构

```
第 N 轮
━━━━━━━━
1. 假设（可能的原因）
2. 验证（收集证据）
3. 结果分析
4. 用户确认（继续/结束/新方向）
```

### 第 1 轮：复现与观察
```
目标：确认 bug 存在，理解基本表现

1. 收集信息
   - 错误信息是什么？
   - 在什么环境下发生？
   - 触发条件是什么？

2. 尝试复现
   - 能否稳定复现？
   - 最小复现步骤是什么？

3. 记录观察
   - 实际输出 vs 预期输出
   - 相关的日志/堆栈
```

### 第 2 轮：假设与验证
```
目标：形成假设，收集证据

1. 提出可能原因
   - 可能是什么导致的？
   - 列出 2-3 个假设

2. 设计验证方法
   - 如何验证每个假设？
   - 需要什么证据？

3. 执行验证
   - 收集日志/断点/调试输出
   - 排除或确认每个假设
```

### 第 3 轮：定位根因
```
目标：找到确切的代码位置

1. 缩小范围
   - 问题在哪个模块？
   - 哪段代码最可疑？

2. 精确定位
   - 使用日志、二分法、断点
   - 找到具体的失败行/调用链

3. 理解代码意图
   - 这段代码想做什么？
   - 实际行为与预期差在哪里？
```

### 第 4 轮：设计与实施修复
```
目标：设计并实施最小修复

1. 设计修复方案
   - 最少改动的方案是什么？
   - 会影响其他功能吗？

2. 实施修复
   - 进行代码修改
   - 保持代码风格一致

3. 验证修复
   - 运行相同测试/步骤
   - 确认问题不再出现
```

### 第 5 轮：回归与预防
```
目标：确保修复完整，防止复发

1. 运行回归测试
   - 相关功能是否受影响？
   - 整体测试套件通过？

2. 添加测试
   - 编写针对此 bug 的测试
   - 确保未来不会重现

3. 记录根因
   - 记录根因分析
   - 检查是否有类似问题
```

## 轮次总结格式

每轮结束时输出：

```markdown
## 第 N 轮总结

### 🔍 假设/观察
- ...

### 📊 证据/结果
- ...

### ✅ 结论
- ...

### 用户决策
[ ] 继续下一轮
[ ] 已经找到原因，进入修复
[ ] 需要尝试其他方向
[ ] 问题无法复现，结束调试
```

## 状态保存

每轮结束后保存到 `workspace/debug-<date>.md`：

```markdown
# 调试记录

> 日期: YYYY-MM-DD
> Bug: <简要描述>
> 轮次: N/5
> 状态: 进行中/已解决/未解决

## 调试历史

### 第 1 轮：...
### 第 2 轮：...

## 当前状态

### 根因假设
- 假设 A: ...
- 假设 B: ...

### 已排除
- ...

### 待验证
- ...

## 修复记录
（修复实施后填写）

## 根因分析
（调试结束后填写）
```

## 终止条件

满足以下任一条件，调试结束：

1. **已解决** — bug 完全修复，测试通过
2. **未解决但已达最大轮次** — 5 轮后仍无法解决，建议升级或寻求帮助
3. **无法复现** — 问题无法稳定复现，无法继续调试
4. **用户决策** — 用户说"够了"或跳过某些步骤

## 输出物

1. **根因分析** — 真正的根本原因
2. **修复方案** — 具体改了什么
3. **回归测试** — 防止复发的测试
4. **预防建议** — 如何避免类似问题

## 常见陷阱

| 想法 | 现实 |
|------|------|
| "看起来像是这个问题" | 需要证据，不是直觉 |
| "先改了再说" | 可能引入新问题 |
| "测试通过了就行" | 可能只是测试没覆盖到 |
| "这个 bug 很简单" | 简单 bug 也需系统化方法 |
