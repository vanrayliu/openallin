# Systematic Debugging — 系统化调试

> 遇到任何 bug、测试失败或意外行为时，在提出修复方案之前使用此技能。

## 核心原则

1. **不要猜测** — 先理解，再修复
2. **复现问题** — 不能复现的 bug 无法修复
3. **定位根因** — 修复症状 ≠ 修复问题
4. **验证修复** — 修复后必须验证

## 工作流

### Step 1: 复现
```
1. 明确复现步骤
2. 确认环境一致性
3. 记录失败输出
```

### Step 2: 定位
```
1. 缩小问题范围
2. 使用二分法或日志定位
3. 找到具体的失败代码行
```

### Step 3: 理解
```
1. 这段代码的意图是什么？
2. 实际行为与预期行为的差异？
3. 是什么导致了这个差异？
```

### Step 4: 修复
```
1. 设计最小修复方案
2. 实施修复
3. 运行测试验证
```

### Step 5: 预防
```
1. 添加回归测试
2. 记录根因分析
3. 检查是否有类似问题
```

## 常见陷阱

| 想法 | 现实 |
|------|------|
| "看起来像是这个问题" | 需要证据，不是直觉 |
| "先改了再说" | 可能引入新问题 |
| "测试通过了就行" | 可能只是测试没覆盖到 |
| "这个 bug 很简单" | 简单 bug 也需系统化方法 |

## 输出物

- 根因分析
- 修复方案
- 回归测试
- 预防建议
