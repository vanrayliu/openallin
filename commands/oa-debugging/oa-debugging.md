# /oa:debugging — 系统化调试

当用户输入 `/oa:debugging <issue>` 时执行此技能。

## 核心设计：多轮迭代

```
┌─────────────────────────────────────────────────────────────┐
│  /oa:debugging 是多轮迭代过程：                             │
│                                                             │
│  假设 → 验证 → 定位 → 修复 → 测试                          │
│    ↑                                      │                │
│    └──────────────────────────────────────┘                │
│                                                             │
│  每轮结束后向用户确认，直到 bug 完全解决                     │
└─────────────────────────────────────────────────────────────┘
```

## 工作流程

### 轮次结构

```
第 N 轮
━━━━━━━━
1. 假设（可能的原因）
2. 验证（收集证据）
3. 结果分析
4. 用户确认（继续/结束/新方向）
```

### 第 1 轮：复现与观察
```
目标：确认 bug 存在，理解基本表现

1. 收集信息
   - 错误信息是什么？
   - 在什么环境下发生？
   - 触发条件是什么？

2. 尝试复现
   - 能否稳定复现？
   - 最小复现步骤是什么？
```

### 第 2 轮：假设与验证
```
目标：形成假设，收集证据

1. 提出可能原因
   - 可能是什么导致的？
   - 列出 2-3 个假设

2. 设计验证方法
   - 如何验证每个假设？
   - 需要什么证据？
```

### 第 3 轮：定位根因
```
目标：找到确切的代码位置

1. 缩小范围
   - 问题在哪个模块？
   - 哪段代码最可疑？

2. 精确定位
   - 找到具体的失败行/调用链
```

### 第 4 轮：设计与实施修复
```
目标：设计并实施最小修复

1. 设计修复方案
2. 实施修复
3. 验证修复
```

### 第 5 轮：回归与预防
```
目标：确保修复完整，防止复发

1. 运行回归测试
2. 添加测试
3. 记录根因
```

## 轮次总结格式

```markdown
## 第 N 轮总结

### 🔍 假设/观察
- ...

### 📊 证据/结果
- ...

### ✅ 结论
- ...

### 用户决策
[ ] 继续下一轮
[ ] 已经找到原因，进入修复
[ ] 需要尝试其他方向
```

## 状态保存

每轮结束后保存到 `workspace/debug-<date>.md`：

```markdown
# 调试记录

> 日期: YYYY-MM-DD
> Bug: <简要描述>
> 轮次: N/5
> 状态: 进行中/已解决/未解决

## 调试历史
### 第 1 轮：...
### 第 2 轮：...

## 当前状态
### 根因假设
- 假设 A: ...
### 已排除
- ...

## 修复记录
（修复后填写）

## 根因分析
（结束后填写）
```

## 终止条件

1. **已解决** — bug 完全修复，测试通过
2. **未解决** — 5 轮后仍无法解决，建议升级
3. **无法复现** — 问题无法稳定复现
4. **用户决策** — 用户说"够了"

## 输出物

1. **根因分析**
2. **修复方案**
3. **回归测试**
4. **预防建议**
