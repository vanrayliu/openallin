# 编码规范

## 通用规则

1. 使用有意义的变量和函数名
2. 函数应该短小，只做一件事
3. 避免深层嵌套（最多 3 层）
4. 优先使用早期返回
5. 保持 DRY 原则
6. 注释解释"为什么"，不是"做什么"

## 文件组织

- 每个文件应该有单一职责
- 相关代码放在一起
- 公共 API 放在文件顶部
- 导入顺序：标准库 → 第三方 → 本地

## 错误处理

- 不要吞掉错误
- 使用具体的错误类型
- 提供有意义的错误信息
- 在边界处处理错误

## 测试

- 测试行为，不是实现
- 测试名描述场景
- 使用 Given-When-Then 结构
- 每个测试只验证一个行为
