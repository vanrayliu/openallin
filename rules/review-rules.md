# 审查规则

## 审查原则

1. 审查代码，不是人
2. 具体而非模糊
3. 建设性而非批评性
4. 区分"必须改"和"建议改"

## 审查清单

### 正确性
- [ ] 逻辑正确
- [ ] 边界条件处理
- [ ] 错误处理完整

### 完整性
- [ ] 所有需求已实现
- [ ] 测试覆盖充分
- [ ] 文档已更新

### 一致性
- [ ] 代码风格一致
- [ ] 命名清晰
- [ ] 架构模式遵循

### 安全性
- [ ] 无安全隐患
- [ ] 输入已验证
- [ ] 无敏感信息泄露

### 可维护性
- [ ] 代码可读
- [ ] 无过度复杂
- [ ] 依赖合理

## 反馈格式

```
[必须改/建议改] <具体问题>
位置: <文件:行号>
原因: <为什么这是问题>
建议: <如何修复>
```
